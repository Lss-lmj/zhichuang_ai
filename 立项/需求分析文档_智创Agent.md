# 智创Agent·计算机学科垂类大模型与双创能力赋能平台

## 需求分析文档

版本：v0.2  
定位：面向“学科垂类大模型与创新应用开发”赛题的需求收敛版  
核心场景：计算机学科助学、课程作业代码分析、教师学情诊断、竞赛创新能力培养

---

## 一、项目定位

### 1.1 赛题理解

本项目对应赛题为“面向一流学科建设的学科垂类大模型与创新应用开发”。赛题要求不是单纯开发一个通用校园平台，而是围绕一个明确学科，构建具备专业知识支撑、自然语言交互、内容可追溯能力的垂类大模型应用，并落到真实教学或科研场景中。

本项目选择的具体学科为：**计算机科学与技术**。

本项目不把目标定义为从零研发或发布一个新的基础模型，而是基于通用大模型能力，结合计算机学科知识库、RAG 检索增强和场景化 Agent 工作流，形成可运行、可验证、可追溯的学科垂类智能体应用。

### 1.2 产品一句话定义

智创Agent·计算机学科垂类大模型与双创能力赋能平台，是一个基于通用大模型的计算机学科垂类智能体应用。系统以计算机学科知识库、竞赛知识库、学生能力画像和 RAG 检索增强为支撑，为学生提供可追溯的学习规划、竞赛准备和组队推荐，同时通过课程作业代码分析为教师提供学情诊断看板和学生作业分析报告。

### 1.3 赛题要求与本项目设计映射

| 赛题要求 | 本项目响应设计 | 优先级 |
|---|---|---|
| 聚焦某一特定学科 | 聚焦计算机科学与技术，覆盖核心课程、实践项目、竞赛与科研方向 | P0 |
| 构建学科垂类智能体应用 | 基于通用大模型 + 计算机学科知识库 + RAG + 场景化 Agent 工作流，不从零训练基础模型 | P0 |
| 贴合真实教学或科研场景 | 聚焦学生学习成长、课程作业代码分析、教师学情诊断和竞赛准备 | P0 |
| 自然语言多轮交互 | 学生和教师均可通过自然语言发起任务，支持上下文追问 | P0 |
| 知识权威可信 | 引入课程大纲、教材目录、竞赛官方通知、教师维护资料等知识源 | P0 |
| 内容可追溯 | 关键结论必须展示引用来源、更新时间和证据片段 | P0 |
| 个性化与自适应 | 基于学生能力画像、目标方向和历史反馈生成个性化建议 | P0 |
| 复杂问题求解 | 支持将“如何准备竞赛/科研/就业”拆解为阶段目标、任务和资源 | P1 |
| 演示与评测验证 | 使用公网 Demo、演示账号、典型测试案例和输出记录验证核心流程 | P0 |
| AI 生成内容标识 | 所有生成式回答明确标注“AI 生成，仅供参考，需结合官方信息核验” | P0 |

### 1.4 产品边界

#### 本版本重点解决

1. 学生不知道如何从课程、项目、竞赛中形成能力成长路径。
2. 学生获取竞赛信息分散，难以判断自己适合参加哪些竞赛。
3. 学生组队依赖随机社交，缺少基于技能和目标的匹配依据。
4. 教师在竞赛指导和课程项目组织中，难以快速了解学生技能结构和候选梯队。
5. 大模型回答计算机学习和竞赛问题时缺少可信来源和上下文适配。

#### 本版本暂不重点解决

1. 不做完整教务系统替代。
2. 不做泛社交平台或校园社区。
3. 不依赖实时接入学校全部真实成绩数据作为前置条件。
4. 不承诺自动判断学生性格、社交兼容性等难以验证的指标。
5. 不把“模型微调”作为必须项，优先保证知识库、RAG、工作流和应用闭环可运行。

## 二、目标用户与核心场景

### 2.1 用户角色

| 角色 | 典型用户 | 核心诉求 | 使用频率 |
|---|---|---|---|

| 教师 | 任课教师、竞赛指导教师、辅导员 | 了解学生能力分布，筛选竞赛梯队，辅助教学改进 | 中频 |

### 2.2 学生核心画像

#### 画像一：低年级探索者

| 属性 | 描述 |
|---|---|
| 基本情况 | 大一或大二，对 AI、算法、开发等方向感兴趣，但不知道如何开始 |

| 核心痛点 | 学习路径碎片化，不知道课程、项目、竞赛之间如何衔接 |
| 期望能力 | 输入目标后，系统能给出阶段化路径、推荐资源和可执行任务 |

#### 画像二：竞赛准备者

| 属性 | 描述 |
|---|---|
| 基本情况 | 大二或大三，想参加蓝桥杯、挑战杯、互联网+、ACM 等竞赛 |

| 核心痛点 | 不知道适合哪个赛道，缺少队友，缺少针对性准备清单 |
| 期望能力 | 系统根据能力画像和竞赛要求推荐竞赛、解释原因，并辅助组队 |

#### 画像三：教师指导者

| 属性 | 描述 |
|---|---|
| 基本情况 | 课程教师或竞赛指导教师，需要组织学生参与课程项目或竞赛 |
| 当前行为 | 依靠成绩、印象和学生主动报名筛选候选人 |
| 核心痛点 | 难以快速了解学生真实实践能力和方向兴趣 |
| 期望能力 | 查看班级作业完成情况、学生代码分析报告和共性能力短板 |

### 2.3 核心业务场景

| 场景编号 | 场景名称 | 用户任务 | 系统输出 |
|---|---|---|---|
| S1 | 计算机学科学习问答 | 学生询问课程概念、方向选择、学习方法 | 带引用来源的分步解释和后续学习建议 |
| S2 | 个性化学习路径规划 | 学生输入目标方向、基础水平和时间约束 | 阶段目标、课程建议、项目任务、竞赛节点 |

| S4 | 竞赛组队推荐 | 学生创建组队需求或寻找队友 | 候选队友、技能互补说明、沟通建议 |
| S5 | 教师作业分析 | 教师查看一次课程作业的班级完成情况和学生代码分析报告 | 分数分布、代码质量分析、共性问题、学生报告 |
| S6 | 知识库维护 | 管理员上传课程资料、竞赛规则、项目案例 | 结构化知识条目、向量索引、引用元数据 |

## 三、业务痛点分析

### 3.1 学生端痛点

| 编号 | 痛点 | 当前后果 | 需求机会 |
|---|---|---|---|
| P-S01 | 学习路径模糊 | 低年级学生盲目刷课或跟风学技术 | 用画像和目标生成个性化路径 |
| P-S02 | 竞赛信息分散 | 错过报名时间，无法理解赛道要求 | 聚合官方信息并自动解读 |
| P-S03 | 竞赛准备低效 | 不知道应该补哪些知识和项目能力 | 根据竞赛要求生成准备清单 |
| P-S04 | 组队随机 | 技能错配、目标不一致、沟通成本高 | 基于技能、目标、时间进行可解释推荐 |
| P-S05 | 能力证明分散 | 简历和项目经历无法形成结构化展示 | 形成学生能力画像和成长记录 |

### 3.2 教师端痛点

| 编号 | 痛点 | 当前后果 | 需求机会 |
|---|---|---|---|
| P-T01 | 学情掌握粗放 | 只看成绩，难以判断实践能力 | 展示技能维度和项目经历分布 |
| P-T02 | 竞赛梯队筛选低效 | 靠学生主动报名和教师印象 | 根据竞赛要求筛选候选人 |
| P-T03 | 教学改进缺少数据 | 难以定位班级共性短板 | 输出班级能力短板和课程建议 |

### 3.3 现有方案不足

| 方案 | 不足 |
|---|---|
| 教务系统 | 偏成绩管理，缺少技能、项目、竞赛、目标方向数据 |
| QQ 群和微信群 | 信息流失快，难检索，缺少结构化匹配 |
| 通用大模型 | 缺少校内课程、竞赛规则和学生画像上下文，回答不可追溯 |
| GitHub 个人主页 | 需要学生主动维护，教师难以批量分析 |
| 普通问卷表格 | 可收集数据，但不能形成智能问答、推荐和持续更新 |

## 四、总体解决方案

### 4.1 总体架构

```text
用户交互层
  学生端：学习问答、路径规划、竞赛推荐、组队推荐
  教师端：班级画像、梯队筛选、学情诊断
  管理端：知识库维护、竞赛信息维护、测试反馈管理

智能体编排层
  学习规划 Agent
  竞赛解读 Agent
  组队推荐 Agent
  教师诊断 Agent
  知识库维护 Agent

模型与检索层
  通用大模型
  计算机学科 RAG
  竞赛知识 RAG
  推荐与匹配规则引擎
  内容安全与引用校验

数据支撑层
  学科知识库
  竞赛信息库
  学生能力画像库
  项目与经历库
  测试评测库
```

### 4.2 核心设计原则

1. 先做可信知识问答，再做复杂推荐。
2. 推荐结果必须可解释，不能只给分数。
3. 学生画像以授权填写和可验证材料为主，避免过度采集。
4. 教师端只展示教学和指导所需信息，严格控制隐私边界。
5. 所有 AI 生成内容需要标识、引用和反馈入口。
6. 比赛交付优先保证可演示、可复现、可验证。

## 五、功能需求

### 5.1 F0：智能交互与可信回答基础能力

#### F0-001 自然语言多轮对话

| 项目 | 内容 |
|---|---|
| 需求描述 | 用户可以用自然语言提出学习、竞赛、组队、教师诊断相关问题，系统支持多轮追问和上下文理解 |
| 用户角色 | 学生、教师 |
| 优先级 | P0 |
| 输入 | 用户问题、用户角色、会话上下文、学生画像或班级范围 |
| 输出 | 回答内容、引用来源、下一步建议、AI 生成标识 |
| 验收标准 | 连续 3 轮追问中保持上下文一致，不将教师问题误判为学生问题 |

#### F0-002 引用来源与可追溯展示

| 项目 | 内容 |
|---|---|
| 需求描述 | 涉及课程知识、竞赛规则、官方时间、导师方向等信息时，必须显示引用来源 |
| 用户角色 | 学生、教师 |
| 优先级 | P0 |
| 输出 | 来源标题、来源类型、更新时间、引用片段或链接 |
| 验收标准 | 典型问题中至少 80% 的事实性结论能够关联到知识库来源 |

#### F0-003 AI 生成内容标识

| 项目 | 内容 |
|---|---|
| 需求描述 | 所有生成式结果显式标注 AI 生成属性和核验建议 |
| 用户角色 | 全部用户 |
| 优先级 | P0 |
| 验收标准 | 问答、推荐、报告、路径规划页面均包含 AI 生成标识 |

### 5.2 F1：计算机学科知识库与 RAG

#### F1-001 学科知识库构建

| 项目 | 内容 |
|---|---|
| 需求描述 | 构建计算机学科专业知识库，为学习问答、路径规划和竞赛准备提供知识支撑 |
| 用户角色 | 管理员、教师 |
| 优先级 | P0 |
| 知识范围 | 围绕算法竞赛、AI 应用开发、软件项目实践三条首批路径，组织课程大纲、核心知识点、实验项目说明、竞赛规则、项目案例和方向资料 |
| 元数据 | 标题、来源类型、所属课程、所属方向、适用年级、发布时间、更新时间、维护人 |
| 验收标准 | 首版至少包含 5 门核心课程、10 条竞赛资料、10 个项目案例或学习资源 |

首批重点路径：

| 路径 | 目标用户 | 知识库重点 | 典型输出 |
|---|---|---|---|
| 算法竞赛 | 想参加蓝桥杯、ACM 等竞赛的学生 | 程序设计、数据结构、算法设计、竞赛规则、训练题单 | 备赛路径、知识短板、训练计划 |
| AI 应用开发 | 想做 AI 项目、创新创业作品的学生 | Python、机器学习基础、数据处理、模型调用、项目案例 | 项目选题、技术路线、学习资源 |
| 软件项目实践 | 想提升工程能力或参加开发类竞赛的学生 | 数据库、软件工程、前后端开发、测试部署、团队协作 | 项目拆解、角色分工、工程规范建议 |

建议首批课程：

| 课程 | 知识点示例 |
|---|---|
| 程序设计基础 | 变量、控制结构、函数、调试、代码风格 |
| 数据结构 | 线性表、树、图、排序、复杂度分析 |
| 算法设计与分析 | 动态规划、贪心、搜索、图算法 |
| 数据库系统 | SQL、范式、事务、索引、查询优化 |
| 软件工程 | 需求分析、设计模式、测试、版本管理 |

#### F1-002 RAG 检索增强生成

| 项目 | 内容 |
|---|---|
| 需求描述 | 用户提问后，系统从知识库检索相关资料，再生成回答，降低幻觉风险 |
| 用户角色 | 学生、教师 |

| 项目 | 内容 |
|---|---|

| 用户角色 | 管理员 |

| 验收标准 | 能查看每条知识的来源、维护人和最近更新时间 |

### 5.3 F2：学生能力画像

#### F2-001 学生画像采集

| 项目 | 内容 |
|---|---|
| 需求描述 | 通过学生授权填写、项目经历、竞赛经历和 GitHub 链接构建基础画像 |
| 用户角色 | 学生 |
| 优先级 | P0 |
| 输入 | 年级、课程基础、技能标签、项目经历、竞赛经历、目标方向、时间投入、GitHub 链接 |
| 输出 | 能力维度评分、标签、证据材料、成长建议 |
| 验收标准 | 学生 5 分钟内可以完成基础画像创建 |

#### F2-002 能力维度模型

| 维度 | 描述 | 证据来源 |
|---|---|---|
| 编程基础 | 语言掌握、代码调试、基础语法 | 自评、项目、课程记录 |
| 算法能力 | 数据结构、复杂度、算法题经验 | 竞赛经历、训练记录、自评 |
| 工程实践 | 项目开发、测试、协作、部署 | GitHub、项目说明、教师评价 |
| AI 与数据能力 | 机器学习、数据处理、模型应用 | 课程、项目、竞赛经历 |
| 学科基础 | 操作系统、数据库、计算机网络等核心课程 | 课程记录、自评 |
| 协作表达 | 文档、汇报、团队协作 | 项目经历、队友评价、教师评价 |

评分口径：

1. 能力评分不是对学生真实能力的绝对判断，也不作为排名或评价学生优劣的依据。
2. 评分本质是“基于证据的相对画像”，用于在学习路径、竞赛推荐和组队推荐场景中判断匹配关系。
3. 每个分数需要同时记录证据来源和置信度。官方成绩、竞赛证书、教师评价、项目代码等可验证证据置信度较高；学生自评可用于冷启动，但置信度较低。
4. 系统展示时优先呈现标签、证据和改进建议，分数只作为辅助排序和推荐解释，不单独作为结论。

#### F2-003 画像可解释与可编辑

| 项目 | 内容 |
|---|---|
| 需求描述 | 学生可查看画像评分依据，并修改自评或补充证据材料 |
| 用户角色 | 学生 |
| 优先级 | P0 |
| 验收标准 | 每个能力标签都能展示至少一个来源或说明，不出现无依据的标签 |

#### F2-004 作业/项目代码分析与能力证据提取

| 项目 | 内容 |
|---|---|
| 需求描述 | 学生可上传课程作业代码、项目压缩包或仓库链接，系统自动分析代码结构、完成度、工程规范和能力证据 |
| 用户角色 | 学生、教师 |
| 优先级 | P0 |
| 输入 | 代码压缩包、仓库链接、作业要求、课程信息 |
| 输出 | 多维度评分、代码结构摘要、问题列表、改进建议、可纳入画像的能力证据 |
| 验收标准 | 系统能生成可解释的代码分析报告，并将证据来源关联到具体文件、模块或代码片段 |

### 5.4 F3：个性化学习路径规划

#### F3-001 目标导向学习路径生成

| 项目 | 内容 |
|---|---|
| 需求描述 | 基于学生当前画像、目标方向和时间约束，生成阶段化学习路径 |

## 六、数据需求

### 6.1 核心数据对象

| 数据对象 | 关键字段 | 用途 |
|---|---|---|
| 用户 | 角色、年级、专业、班级、权限 | 登录、权限控制、个性化 |
| 学生画像 | 技能标签、能力维度、目标方向、证据来源 | 路径规划、竞赛推荐、组队推荐 |
| 知识条目 | 标题、正文、来源、课程、方向、更新时间 | RAG 检索和引用 |
| 竞赛信息 | 名称、主办方、赛道、时间、规则、链接 | 竞赛推荐和准备计划 |
| 组队需求 | 竞赛、方向、角色缺口、时间要求 | 队友匹配 |
| 评测记录 | 测试场景、输入、输出、引用、评分、问题记录 | 技术验证和迭代 |

### 6.2 数据来源优先级

| 优先级 | 数据来源 | 说明 |
|---|---|---|
| 最高 | 官方竞赛通知、课程大纲、教师维护资料 | 用于事实性回答和引用 |
| 高 | 学生授权填写的项目、竞赛和目标信息 | 用于个性化推荐 |
| 中 | GitHub 仓库、公开项目链接 | 用于辅助判断工程实践能力 |
| 低 | 学生自评 | 可用于冷启动，但需要明显标注来源 |

### 6.3 权限与隐私

1. 学生个人画像默认仅本人可见。
2. 进入组队推荐池需要学生主动授权。
3. 教师仅能查看课程、班级或指导范围内的统计信息和授权学生信息。
4. 教师查看学生作业报告时，系统按课程、班级和角色权限控制可见范围。
5. 系统面向学校真实使用，支持真实课程数据在授权范围内展示；公网 Demo 使用演示账号和示例课程数据。

## 七、技术路线建议

### 7.1 大模型应用方式

| 能力 | 建议实现 |
|---|---|
| 基础对话 | 调用通用大模型、开源大模型、OpenAI-compatible 服务或本地模型 |
| 学科增强 | 通过计算机学科知识库 RAG 注入上下文 |
| 竞赛规则问答 | 通过竞赛信息库 RAG 生成带来源回答 |
| 路径规划 | 大模型负责任务拆解，规则引擎负责约束和模板校验 |
| 组队推荐 | 规则评分 + 大模型解释，不完全依赖大模型自由生成 |
| 教师诊断 | 统计分析 + 大模型生成诊断摘要 |

### 7.2 RAG 流程

```text
用户问题
  -> 意图识别
  -> 查询改写
  -> 向量检索 + 关键词检索
  -> 结果重排序
  -> 来源可信度过滤
  -> 大模型生成
  -> 引用校验
  -> AI 标识与风险提示
  -> 评测记录
```

### 7.3 推荐逻辑

#### 竞赛推荐评分

```text
竞赛匹配度 =
  目标方向匹配 * 0.30
  + 技能要求匹配 * 0.30
  + 年级与时间匹配 * 0.15
  + 项目经历相关性 * 0.15
  + 竞赛经验相关性 * 0.10
```

#### 队友推荐评分

```text
队友匹配度 =
  技能互补度 * 0.35
  + 目标一致性 * 0.25
  + 时间可用性 * 0.20
  + 项目经历相关性 * 0.15
  + 竞赛经历相关性 * 0.05
```

评分结果只作为排序依据，系统必须同时展示推荐理由和不确定性说明。

## 八、非功能需求

### 8.1 可用性

| 指标 | 要求 |
|---|---|
| 学生画像创建 | 基础画像填写不超过 5 分钟 |
| 问答响应 | 常规问题 10 秒内返回首轮结果 |
| 路径规划 | 30 秒内生成可读的阶段计划 |
| 教师看板 | 支持按班级、年级、方向筛选 |

### 8.2 准确性与可信度

| 指标 | 要求 |
|---|---|
| 事实性回答 | 关键事实必须带来源 |
| 竞赛时间和规则 | 以官方资料或教师维护资料为准 |
| 不确定问题 | 明确提示无法确认，不编造 |
| 推荐解释 | 每个推荐结果至少包含 2 条理由和 1 条风险提醒 |

### 8.3 安全与合规

1. 系统通过角色权限控制学生、教师、管理员的数据访问范围。
2. 不生成虚假学术成果、虚假竞赛经历或虚假证明材料。
3. 不自动替学生完成违反学术诚信的作业、论文或申报材料。
4. 所有生成内容均标注 AI 生成。
5. 用户可删除或撤回自己的画像和组队授权。

### 8.4 可复现性

1. 提交作品代码、部署说明、模型或模型 ServiceID。
2. 提供知识库样例数据。
3. 提供 3 个以上典型测试问题和输出记录。
4. 提供演示账号和 Demo 访问说明。

## 十二、风险与应对

| 风险 | 表现 | 应对策略 |
|---|---|---|
| 功能范围过大 | 所有模块都做但都不完整 | MVP 聚焦知识库、画像、路径、竞赛、组队、作业代码分析和教师看板 |
| 数据接入困难 | 教务成绩或真实学生数据难获取 | 首版支持手动导入课程、作业和学生提交数据 |
| 大模型幻觉 | 编造竞赛时间、课程要求或资源 | RAG 引用、来源校验、不确定提示 |
| 推荐不可解释 | 用户不知道为什么被推荐 | 每条推荐必须显示依据、短板和风险 |
| 隐私争议 | 学生画像被教师或同学过度查看 | 权限控制、授权加入推荐池、默认隐藏联系方式 |
| 赛题偏离 | 被评为普通组队平台 | 强化学科知识库、RAG、教学场景、可追溯和效果验证 |

---

## 十三、创新点总结

1. 将计算机学科知识库、竞赛知识库和学生能力画像结合，形成面向创新能力培养的垂类智能体。
2. 从“问答”扩展到“诊断、规划、推荐、组队”的多步骤任务闭环。
3. 推荐结果不是黑箱分数，而是展示能力依据、来源证据、短板和下一步行动。
4. 同时服务学生个性化成长和教师课程作业学情诊断，能够形成真实教学场景验证。
5. 全流程加入 AI 生成标识、引用追溯和隐私授权，符合比赛合规要求。

---

## 十四、已定实施口径

1. 首批知识库资料需求包括：课程大纲、课程作业要求、评分 Rubric、代码样例、竞赛规则、项目案例和常见问题。
2. 数据策略：系统面向学校真实使用，支持真实课程数据在授权范围内展示；公网 Demo 使用演示账号和示例课程数据。
3. 首版重点竞赛清单由系统内置并维护更新，管理员可在后台修订竞赛信息。
4. Demo 直接部署在公网，提供演示账号、访问地址和使用说明。
5. 用户访谈、试用记录等线下参赛材料不纳入系统功能范围。
