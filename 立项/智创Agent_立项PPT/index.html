定位决定边界
这里要和“垂类大模型”表达做切割：我们不是造一个新基础模型，而是做基于大模型的学科垂类智能体应用。
痛点

Problem

三类用户，三组高频问题

学生端

  • 不知道课程、项目、竞赛如何衔接
  • 项目能力难以结构化展示
  • 组队依赖群聊和随机匹配

教师端

  • 批量了解学生项目成本高
  • 难以快速定位班级共性问题
  • 只能凭成绩或印象判断实践能力

管理端

  • 竞赛、课程、项目资料分散
  • 缺少统一知识库和引用追溯
  • 缺少可复现的测试评测记录
痛点来自教学过程
这一页把系统必要性讲出来：不是为了炫技，而是解决教学过程里每天会遇到的成本问题。
目标

Goals

两条核心闭环

学生项目闭环

项目 -> 画像 -> 路径 -> 竞赛 -> 组队

基于项目证据、学生画像、目标方向和知识库，为学生生成学习路径、竞赛建议和队友推荐。

教师学情诊断闭环

项目提交 -> 智能分析 -> 报告生成 -> 班级看板

围绕课程项目、竞赛作品、个人作品和双创项目，系统自动生成多维度分析，教师直接查看班级和个人报告。

核心价值：学生知道下一步怎么学，教师知道班级真实问题在哪里。
从单点功能到教学闭环
这一页是整份 PPT 的主心骨。后面所有技术架构都围绕这两个闭环展开。
核心场景一

Project Intelligence

一个学生项目，自动形成学情证据

上传项目代码压缩包、仓库链接、项目说明、评分 Rubric。
结构解析识别语言、依赖、目录、入口文件和主要模块。
多 Agent 分析功能完成度、代码质量、工程规范、能力证据。
生成报告多维度评分、问题定位、亮点、改进建议。
教师查看班级分布、共性问题、单个学生项目报告。
学生项目分析是核心教学场景
这一页回应“学校真实使用”的诉求。系统不是替教师打分，而是把项目材料变成可解释的结构化报告。
报告结构

Scoring Dimensions

项目报告看什么

功能完成度
对照项目目标检查核心功能、运行入口和关键路径
完成度分数、缺失功能、运行风险
代码结构
分析目录组织、模块边界、依赖关系和可读性
结构摘要、模块图、复杂度提示
工程规范
检查 README、依赖文件、运行说明、测试或样例
规范得分、缺失文档、运行建议
能力证据
提取可支撑学生画像的课程知识点和工程能力
证据片段、对应能力、置信度
改进建议
从功能、质量、文档、测试四个方面提出下一步动作
问题清单、优先级、示例修正方向
评分必须可解释，可回溯到文件或片段
注意强调：评分不是绝对能力判断，而是基于提交物证据的相对分析。
核心场景二

Student Growth

学生端不是信息墙，是项目成长工作区

项目资产课程作业、竞赛作品、个人作品、双创项目。
能力画像技能标签、项目经历、分析证据、目标方向。
学习路径阶段目标、课程资源、项目任务、检查点。
竞赛推荐匹配竞赛要求、准备清单、引用来源。
组队推荐基于授权画像、技能互补、目标一致。
阶段反馈根据完成情况和新证据更新建议。
学生端形成个性化成长闭环
学生端要讲“伴随式”和“证据驱动”。项目分析会反哺能力画像。
教师端

Teacher Dashboard

教师直接看到班级项目情况

班级概览

提交率、分数分布、完成度分布、异常提交提示。

共性问题

高频错误、薄弱知识点、工程规范缺口、测试缺失。

学生报告

单个学生的代码结构、质量问题、能力证据和改进建议。

教师直接查看系统生成的学情分析结果，用于课程讲评、项目指导和竞赛梯队建设。
教师学情诊断是核心教学场景
这一页强调教师端是核心教学场景：教师查看报告和班级画像，不承担模型输出确认流程。

Agent 编排

LangGraph Workflow

为什么需要图工作流

CodeAnalysisGraph

代码分析长任务

  • 文件解析 Agent
  • 结构分析 Agent
  • 质量评估 Agent
  • 能力证据 Agent
  • 报告生成 Agent
LearningPlanGraph

学习规划长任务

  • 目标澄清
  • 知识库检索
  • 计划生成
  • 计划评估
  • 保存与阶段反馈
LangGraph 负责长任务状态、节点编排和中断恢复；RAG、评分、权限、引用校验仍然由确定性服务承担。
长任务不是一次对话能解决的
这一页可以解释技术取舍：不是为了用 LangGraph 而用，而是代码分析、多 Agent 和报告生成本来就是图工作流。
